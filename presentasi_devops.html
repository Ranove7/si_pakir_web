Mengapa DevOps Dibutuhkan?

Permasalahan umum dalam pengembangan sistem parkir konvensional:

01

Deployment Manual & Rawan Error

Proses deploy dilakukan secara manual sehingga memakan waktu dan berisiko terjadi inkonsistensi konfigurasi antar environment pengembangan dan produksi.

02

Tidak Ada Monitoring Real-Time

Tidak tersedia sistem pemantauan performa server (CPU, RAM, Disk, Network) sehingga sulit mendeteksi bottleneck atau kegagalan layanan sejak dini.

03

Aplikasi Tidak Terisolasi

Tanpa containerisasi, dependency antar komponen dapat konflik dan menyebabkan sistem tidak stabil atau sulit direplikasi di lingkungan produksi.

🎯 Tujuan

Target Implementasi

Empat tujuan utama yang dicapai dalam proyek DevOps ini:

🐳

Containerisasi Penuh

Seluruh komponen (DB, Backend, Frontend, Monitoring) dikemas dalam Docker container yang terisolasi dan reproducible di environment manapun.

⚡

Otomasi CI/CD

Setiap push ke branch main secara otomatis men-trigger pipeline GitHub Actions untuk build dan deploy ke VPS production tanpa intervensi manual.

📊

Monitoring Real-Time

Prometheus mengumpulkan metrics dari semua service. Grafana menampilkan dashboard visualisasi performa server secara live dan interaktif.

🔒

Akses Terpusat via Nginx

Nginx sebagai reverse proxy merutekan semua trafik melalui port 80, menghindari keterbatasan firewall VPS dan menyederhanakan akses publik.

🔧 Environment Variables

Semua konfigurasi sensitif (DB password, Secret Key, JWT) dikelola via file .env yang di-exclude dari Git.

💾 Persistent Volumes

Data MySQL dan Grafana tersimpan di Docker volume — tidak hilang saat container di-restart atau di-rebuild.

🌐 Isolated Network

Semua container dalam si_parkir_net. Service berkomunikasi via nama container, bukan IP dinamis.

🔄 Auto Restart

Semua service dikonfigurasi restart: always untuk recovery otomatis jika terjadi crash atau reboot VPS.

💡 Kenapa Semua Harus Lewat Port 80?

VPS hanya membuka port 22 (SSH) dan 80 (HTTP). Port 3000, 8000, 9090 tidak bisa diakses dari internet. Nginx menjadi single entry point yang mendistribusikan setiap request ke service yang tepat berdasarkan path URL.

🔧 Kendala & Solusi

Tantangan dalam Implementasi

❌ Kendala
Bcrypt crash di Python 3.11 — library passlib tidak kompatibel dengan bcrypt versi baru
→
✅ Solusi
Migrasi dari passlib ke library bcrypt native langsung tanpa wrapper abstraksi
❌ Kendala
Port 3000, 8000, 9090 diblokir firewall VPS — monitoring tidak bisa diakses publik
→
✅ Solusi
Nginx reverse proxy merutekan semua akses melalui port 80 dengan sub-path routing
❌ Kendala
Grafana "No Data" — datasource URL tidak menggunakan subpath /prometheus
→
✅ Solusi
Update datasource ke http://prometheus:9090/prometheus sesuai external-url config
❌ Kendala
WebSocket URL crash di browser karena hardcoded URL tidak berfungsi di VPS
→
✅ Solusi
URL dibangun dinamis menggunakan window.location.host di React
